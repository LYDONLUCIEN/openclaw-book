## 6. 中国移动场景畅想：Agent如何走进我们的工作

前面章节我们把Agent的技术原理、能力边界、工具生态都过了一遍。现在来到最关键的一章——这些能力，到底怎么落到我们每天的工作里？

我给大家讲三个必须把握的要点，然后穿透四个真实场景。这三个要点搞明白了，Agent在中国移动的落地就不会走弯路。

---

### 6.1 模型底座：内网环境下的"大脑"选择

Agent的"智商"是谁决定的？不是界面漂不漂亮，是底层的大模型。Agent就像一个入职的高材生，它的天花板取决于受过什么训练。而在中国移动的内网环境里，还有一个硬约束：**我们无法直接调用OpenAI、Claude等海外公网模型**，连部分国内公网大模型服务在内网也无法直接访问。

这意味着，我们只能使用自有模型。**没有模型底座，再好的Agent框架都是空中楼阁。**

中国移动的答案是**九天大模型**——全栈自主可控的"大国重器"[^39^]。从数据构建、预训练、微调到推理部署，全链路核心技术自主掌握。九天适配了11个厂家17款国产AI芯片，支持"云边端"灵活部署，2024年10月发布了30款自研行业大模型[^54^]，在视觉理解、多语言理解、编码和数学等方面达到国际领先水平。

同一个Agent框架，接不同的模型，表现天差地别。底层模型强，Agent能准确理解专业术语、生成有深度的分析；底层模型弱，连基站告警信息的含义都可能理解错。

好消息是，中国移动不止有九天。2026年5月发布的**MoMA模型服务平台**，整合超300款业界主流AI模型[^26^][^27^]，独创模型联邦与智能路由技术，单位Token成本压降30%以上。内网环境下通过MoMA就能调用丰富的模型能力，还有基于硬件隔离的"机密模型"服务保障数据安全。

**落地建议：九天大模型 + 聚智智能体平台 → OpenClaw/Hermes接入 → 形成完整Agent能力链。** 九天提供"智商"，聚智平台提供运行环境和组件生态，OpenClaw或Hermes提供灵活的Agent框架。数据全程不出域，安全可控。

【展开讲】九天大模型的训练规模达到超2304台服务器、1.8万张AI加速卡[^39^]，算网大脑4.0实现日调度百万亿级Token、纳管80+EFlops算力资源[^88^][^90^]。这意味着中国移动的模型底座不是"勉强能用"，而是足以支撑集团级Agent规模化部署。

---

### 6.2 Skills/Pip源：内网环境的"弹药库"

Agent的能力来自Skills，就像手机的App。但内网环境有一个特殊挑战：**无法访问公网的ClawHub、PyPI、npm等公共仓库。**

好比买了一部新手机，应用商店却打不开——Agent框架再强大，没有Skills就是"有手无指"。

解决思路有两个。**第一，搭建内部Skills仓库**，定期同步公网优质社区Skills，经安全审核后导入内网。聚智平台本身就有统一组件仓库设计，所有组件在平台内沉淀汇聚[^55^]。**第二，搭建内部Pip/Npm镜像源**，定期同步公网依赖包，经安全审计后供内网使用[^113^]。

聚智平台有一个很大的先发优势——**内置了150+行业Skills**（MobileClaw框架）[^26^]。这些不是"helloworld"级别的通用技能，而是覆盖通信、运维、市场、客服等领域的标准化技能包，可以直接作为Agent的能力库使用。

最终形成三条腿走路的内网Skills生态：**自有Skills为主 + 公网优质Skills为辅 + 内部按需自建**，Agent的能力就不会受限于网络隔离。

【展开讲】OpenClaw的ClawHub注册了近14,000个Skills，但超过50%被判定为垃圾、重复或低质量，396个被标记为恶意。因此内部镜像源必须配合严格审核——每个Skills要经过源码审查、功能测试、安全扫描三关，确保不给内网引入风险。

### 6.3 浏览器自动化：打通内部系统的"万能钥匙"

这是中国移动最痛的痛点之一。IT系统经过几十年建设，**内部有数千个异构系统**，很多老系统只能通过浏览器登录操作，没有API接口[^29^]。

大家每天的场景可能是：打开浏览器 → 登录OA → 复制数据 → 切换到另一个系统 → 重新登录 → 粘贴数据 → 再切换到第三个系统……几十套系统各自独立，反复登录切换，大量时间耗在机械操作上。

Agent加上浏览器自动化，就能解决这个问题：**Agent控制浏览器 → 自动登录 → 读取数据 → 操作界面 → 汇总结果。**

磐匠RPA的浏览器自动化能力可以模仿人类操作——自动打开浏览器、访问登录页、输入账号密码、导航到指定页面、提取数据、导出Excel[^51^][^66^]，全程无需人工干预，7×24小时运行。

磐匠从2019年开始建设，经历了三个阶段[^117^][^172^]：

| 阶段 | 时间 | 核心技术 | 能力 |
|------|------|----------|------|
| RPA | 2019-2022 | 流程自动化、规则引擎 | 固定规则执行 |
| IPA | 2022-2024 | RPA+AI+NLP+OCR | 非结构化数据处理 |
| AI Agent | 2024至今 | 大模型+RPA+IDP+GSUT | 自适应界面变化 |

最关键的升级方向是**从固定规则RPA转向AI Agent驱动的智能自动化**。传统RPA需要精确配置每个步骤的元素位置，界面改版就要重配。而融合大模型和GSUT（通用语义理解技术）的磐匠智能体能**自主识别界面、自主学习操作**[^173^]，界面变了也能自适应调整。磐匠的"观摩式"无代码生成更是一大亮点——用户操作一遍业务，磐匠就能自动生成智能体[^173^]。

目前磐匠已上岗**17000+数字员工**，年减负**70万人天**，覆盖16大领域[^49^]。

### 6.4 场景穿透1：日常办公自动化

每天早上面对几十封新邮件，逐个阅读、分类、提取待办、安排工作，平均耗掉30分钟。

用Agent的流程是：**Agent登录邮箱 → 分类邮件 → 提取关键信息 → 识别待办 → 生成计划文档 → 发送到飞书。**

具体展开：Agent自动分类工作邮件、通知公告、审批流转、订阅信息；提取每封邮件核心内容并摘要；识别"需要回复""需要处理""需要参会"等任务；按优先级生成结构化工作计划；通过飞书Bot推送给员工确认。

广西移动的AI智能办公助手已实现类似的"问写办查"能力，覆盖16类知识库、6万+知识点，月均使用量超100万次[^126^]。

**预期效果：从30分钟/天 → 3分钟/天。** 员工看一眼Agent生成的计划，微调优先级即可。

### 6.5 场景穿透2：月度经营分析报告

省公司市场部经理每月需要从8个内部系统提取数据，制作20页PPT。传统方式**耗时2-3天**。

Agent流程：**自动登录各数据平台 → 提取指标 → 清洗数据 → 生成图表 → 撰写分析 → 排版为PPT。**

具体展开：Agent接收指令后调用磐匠RPA，依次登录8个系统导出数据[^51^]；自动合并多源数据，统一格式；计算同比/环比，生成可视化图表；调用九天大模型自动生成趋势分析、异常预警和结论建议；最终输出20页完整PPT + Excel数据底表 + 关键发现摘要。员工确认内容、微调细节后提交领导。

甘肃移动在类似数据处理场景中已实现分钟级处理[^191^]。

**预期效果：从2-3天 → 30分钟，效率提升90%以上。**

### 6.6 场景穿透3：网络故障智能诊断

地市运维中心收到批量告警，基站性能下降。传统方式需手动登录多个网管系统、电话咨询专家，**耗时2-4小时**。

Agent流程：**接收告警 → 登录网管系统 → 查询日志 → 关联分析 → 定位根因 → 生成诊断报告 → 自动派单。**

具体展开：告警触发运维Agent → 关联分析聚类去重（自动驾驶网络框架已实现**1000:1告警压缩比**[^47^]）→ 调用知识图谱匹配300类故障场景 → 自动登录网管系统采集多维数据、交叉分析定位根因 → 输出故障根因、影响范围、处置建议、优先级 → 自动派单、发送通知、启动计时 → 维护人员现场处置 → Agent闭环验证。

湖南移动SPN网络AI故障运维智能体已投入实战，大模型在线注入专家经验，思维链技术实时生成诊断流程[^59^]。广东移动Agentic AI方案实测故障分析时长减少47%，排障提效约30%[^68^]。

**预期效果：从2-4小时 → 10分钟，故障诊断自动化率从60% → 90%。** 重要故障30分钟内清零，这是浙江移动已验证的效果[^110^]。

### 6.7 场景穿透4：跨系统数据稽核

财务部每月底需核对5个系统的用户数据一致性——ERP、报销、银行、合同、资产系统。传统方式逐个导出、手工比对，**耗时5个工作日**。

Agent流程：**依次登录各系统 → 导出数据 → 自动比对 → 标识差异 → 生成稽核报告。**

具体展开：财务人员发出指令 → Agent浏览器自动化登录5个系统，导出数据 → 多源数据自动关联匹配，金额、日期、科目逐项比对 → 差异自动标记分类（金额差异、时间差异、科目不符）→ 生成稽核报告（总单据数、正常/异常笔数、差异明细）→ 自动发送预警邮件 → 财务人员仅需复核异常项。

甘肃移动财务稽核数字员工月均处理超5000笔单据，年均节约人工时长超1000人天，从数小时压缩至分钟级[^40^][^41^]。

**预期效果：从5个工作日 → 1小时，数据准确率提升至99.9%以上，人工工作量减少90%。**

### 6.8 Agent不是替代，是增强

Agent这么能干，会不会取代我们？**不会。Agent是增强，不是替代。**

#### "碳硅并举"

中国移动提出了精准的理念——**"碳硅并举"**[^41^]。碳基员工（人类）负责创造性、决策性、情感性工作；硅基员工（Agent）负责重复性、规则性、计算性工作。人机协同，各取所长。Agent不会写战略规划，不会处理需要人情味的关系协调，但可以把让你每天加班的重复劳动接过去。

#### 与现有系统的关系

Agent和聚智平台、磐匠RPA的关系是**"大脑升级"，不是"推倒重来"。** 磐匠17000+数字员工已在运行[^49^]，350+场景已落地[^40^]。Agent不是要换掉已有投资，而是给它们装上更聪明的大脑。RPA负责"动手"，Agent负责"动脑"。

| 维度 | RPA单独 | RPA + AI Agent |
|------|---------|----------------|
| 决策能力 | 基于固定规则 | 大模型智能决策 |
| 适应性 | 流程变更需重配 | 自动感知并调整 |
| 处理复杂度 | 结构化数据为主 | 结构化+非结构化 |
| 人机交互 | 无，纯自动执行 | 自然语言交互，人机协同 |

#### 落地路径建议

四步走：**①试点场景选择**（高频、重复、规则明确）→ **②小范围验证**（一个场景、一个Agent、跑通全流程）→ **③沉淀Skills**（固化成可复用资产）→ **④逐步推广**（从一个场景到十个场景，从一个部门到多个部门）。甘肃移动从试点到350个场景落地，就是这样一步一步走出来的[^40^]。

## 6D. 补充：Agent落地的组织保障

【展开讲】技术问题解决后，还有三个组织层面的保障要考虑。没有这些保障，Agent项目很容易沦为"技术盆景"——看着漂亮，用不起来。

### 人才培养

Agent落地需要**"Agent指挥官"**这个新角色——不需要会编程，但需要三种核心能力：**会描述需求**（让Agent清楚知道要做什么）、**会分解任务**（把复杂工作拆成Agent能执行的步骤）、**会评估结果**（判断Agent的输出是否合格、哪里需要改进）。

这就像管理一个虚拟下属——你不需要亲自干他的活，但你得知道怎么给他下指令、怎么检查他的工作。建议每个部门至少培养1-2名Agent指挥官，参加聚智平台的低代码/无代码培训。聚智平台的设计初衷就是让不懂代码的一线业务人员也能用上AI[^28^]。可以先从年轻员工中选拔，他们对新技术接受快，稍加培训就能胜任。

### 安全与合规

Agent能操作系统、能访问数据，还能自动发邮件、派工单——权限这么大，安全怎么保障？

**数据安全方面**，Agent的所有操作日志必须全程审计，每一步干了什么、访问了哪些数据、生成了什么结果，都要有完整记录，事后可追溯。**敏感操作人工确认**，修改数据、派发工单、发送邮件等关键环节必须设置人工确认环节，Agent不能擅自做主。这相当于给Agent装了一个"紧急刹车"。

**权限管控方面**，坚持最小权限原则——每个Agent只给它完成任务所必需的权限，不多给。Skills设白名单，未经审核的Skills不允许安装使用。操作范围要限定，这个Agent只能访问哪些系统、只能操作哪些数据，边界必须划清楚[^29^]。聚智平台本身就有全生命周期安全管控机制，可以充分利用。

### 成本管理

几万个Agent同时运行，Token消耗会非常大。如果规划不好，算力成本可能成为一个"隐形黑洞"。

核心思路有三条。**优先使用内部部署模型**，九天和MoMA的单位Token成本比公网API低30%以上[^27^]，而且内网调用更安全。**合理规划模型调用策略**，简单任务（如格式转换、数据比对）用小模型，复杂任务（如分析推理、报告撰写）才用大模型，不要杀鸡用牛刀。**做好用量监控和限额**，每个Agent设置Token使用上限，防止因陷入无限循环或遭受攻击导致Token爆量。聚智平台的智能路由技术可以在这方面发挥重要作用，自动选择性价比最高的模型路径。
