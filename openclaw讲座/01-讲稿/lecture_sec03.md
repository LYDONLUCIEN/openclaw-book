## 3.1 OpenClaw是什么：一句话理解

如果说前面我们讲的是Agent的"通用原理课"，那么从这一章开始，我们要认识一位真正的"明星学生"——OpenClaw。

它是什么？用一句话概括：**开源、自托管的AI Agent操作系统。** 装在你自己的服务器上，通过飞书、钉钉、微信这些你天天用的工具跟它对话，让它帮你做事——查邮件、管日程、操作浏览器、写代码、整理报告，甚至可以控制你家里的智能设备。[^1^]

大家想想这个区别：ChatGPT是一位"顾问"，你问它答，说完就完；OpenClaw是一位"员工"，你交代任务，它自己规划、自己执行、给你结果。更关键的是——**这位员工完全在你家办公，不归任何公司管。**

我们来看看它创下的纪录，这组数据截至2026年3月8日：[^1^]

| 指标 | 数据 |
|------|------|
| GitHub Stars | **278,932** —— 全球软件项目第一，已超越React |
| ClawHub注册Skills | **13,729** 个 |
| 内置Skills | **55** 个，开箱即用 |
| 支持消息渠道 | **20+** 平台（飞书、钉钉、企微、QQ、Telegram等）|
| 代码贡献者 | **1,075+** 人 |
| 项目年龄 | 不到 **4个月** |

不到4个月从零冲到27.9万Stars——什么概念？React用了超过10年才达到23万Stars。这是开源历史上前所未有的增长速度。2026年3月3日，OpenClaw正式超越React登顶GitHub全球第一的那一天，平均每10秒就有一个开发者点下Star。[^1^]

【展开讲】OpenClaw的诞生本身就带着传奇色彩。2025年11月，奥地利开发者Peter Steinberger把它作为一个周末项目发布，取名ClawdBot——Claw是"爪子"，致敬Anthropic的Claude。2026年1月中旬，项目突然爆发：72小时内获得6万Stars，某天单日增长9,000个。1月27日，Anthropic发来商标警告，说名字跟Claude太像，Peter被迫三天内两次改名，最后定名OpenClaw——强调开源属性，同时保留了龙虾吉祥物。2月14日情人节，Peter宣布加入OpenAI，Sam Altman亲自发推称他为"天才"。项目移交开源基金会运营，至今保持近乎每日更新的节奏。[^1^]

## 3.2 架构设计：Gateway-Node-Channel三层架构

OpenClaw的架构设计非常清晰，我用人来打比方——

### Gateway（网关）：大脑

Gateway是OpenClaw的**中央大脑**，24小时不间断运行，默认监听本地的`ws://127.0.0.1:18789`。所有消息进来先到Gateway，由它做路由、调度、管理Session。一台主机只运行一个Gateway实例——因为WhatsApp等渠道需要独占会话，多实例会冲突。[^1^]

### Node（节点）：手脚

如果说Gateway是大脑，那Node就是**手和脚**——负责在本地设备上实际执行任务。操作浏览器、运行Shell命令、控制摄像头录屏、读写文件，这些都是Node的活。Node通过WebSocket连接到Gateway，接收指令、返回结果。[^1^]

### Channel（渠道）：五官

Channel是OpenClaw连接外部世界的**五官**——通过它，Agent能听到你说话（接收消息）、也能回应你（发送回复）。目前支持的渠道超过20个：[^1^]

| 类型 | 代表平台 |
|------|---------|
| 国际主流 | Telegram、WhatsApp、Discord、Slack、Signal |
| 国内企业 | **飞书、钉钉、企业微信、QQ** |
| 特定场景 | iMessage、微信个人号（需中转）、Google Chat |

飞书和QQ是国内用户接入最活跃的渠道。QQ是腾讯官方开放了Bot能力，扫码1分钟就能绑定；飞书自OpenClaw 2026.2版本起获得原生内置支持，是国内企业场景的首选。[^1^]

### 安全设计：默认只监听本地

这里要特别强调OpenClaw的一个安全设计——**Loopback-First**。Gateway默认只绑定`localhost`（127.0.0.1），不开放任何外网端口。这意味着什么？

你的Agent默认只在本地网络可访问，外面的人根本找不到它。需要远程访问时，官方推荐通过Tailscale Serve或Funnel暴露，而不是直接开端口。这是一个"默认即安全"的设计思路，对安全意识强的企业特别友好。[^1^]

【展开讲】一条消息从发出到回复的完整路径是这样的：用户在飞书上发消息 → Channel接收 → Gateway路由判断 → Agent处理思考 → Node执行具体操作 → 通过Channel回复用户。Gateway作为守护进程（daemon）持续运行，不像命令行工具那样会话结束就丢失上下文。它会不断积累记忆，用得越久越懂你。[^1^]

## 3.3 四层记忆系统：为什么OpenClaw"记得住"

好，接下来这一节是**OpenClaw最核心的差异化优势**，也是我个人最欣赏它的地方——四层记忆系统。

大家有没有这种感觉：你跟ChatGPT聊天，聊久了它就开始"忘事"——前面的要求后面就不认了。这是因为大模型有"上下文窗口"限制，对话太长，早期的内容就被挤掉了。

OpenClaw是怎么解决这个问题的？它设计了一套**四层记忆架构**，从永久的人格定义到实时的对话上下文，层层递进。用一个比喻：这就像一个真实的人——有灵魂、有工具箱、有个人经历、有当下对话。[^1^]

### SOUL（灵魂）：你是谁

SOUL.md文件定义Agent的**人格和价值观**——"你是谁、你相信什么、你的行为准则"。这个文件创建后**永久不变**，是Agent的身份内核。你可以在里面写："你是一位严谨的财务分析师，所有数据必须标注来源"，或者"你是一位轻松的创意写手，喜欢用幽默的方式表达"。Agent会始终遵循这个人格设定。[^1^]

### TOOLS（工具）：你能做什么

TOOLS层是当前可用的Skills列表，存储在`skills/`目录下。Agent在启动时会扫描三层Skill来源——项目级、用户级、内置级——构建出自己"当前掌握的技能清单"。这一层是**按需加载、动态变化**的，安装新Skill就多了能力，卸载就少掉。[^1^]

### USER（用户记忆）：越用越懂你

这是最有温度的一层。USER记忆存储在MEMORY.md和向量数据库中，记录**关于你的偏好、历史决策、重要事实**。比如你告诉它"我喜欢在周报里加数据图表"、"我是左撇子所以鼠标在左边"、"我偏好用飞书而不是邮件"——这些都会被Agent写入MEMORY.md，以后每次对话都会参考。[^1^]

更妙的是，OpenClaw默认启用了**语义记忆搜索**——结合Embedding向量相似度和BM25关键词匹配两种策略。你不用精确描述需求，说一句"之前讨论过的那个部署问题"，Agent就能从记忆中找到相关内容。[^1^]

### Session（会话）：当下对话

最外层是Session级别的实时上下文，存在内存和sessions.json中。这是Agent"现在正在想什么"的短期记忆。当对话太长、接近token上限时（默认阈值约4000 tokens），OpenClaw会触发一个**Pre-Compaction**机制——[^1^]

这个过程完全静默，用户感知不到。Agent在后台自动把重要记忆压缩写入MEMORY.md和Daily Log，释放上下文空间。这意味着：**即使对话超长，关键信息也不会随着上下文窗口的滑动而丢失。**

### 自动记忆保存：不会"忘记"

OpenClaw还有一个Daily Logs设计——每天的交互记录以追加方式写入`memory/YYYY-MM-DD.md`文件。Session开始时，Agent会自动读取今天和昨天的日志，保证对话的连续性。[^1^]

对比一下：Claude Code等工具的会话结束后上下文就消失了，下次再开是全新的对话。而OpenClaw通过文件系统实现了**真正的持久记忆**——你跟它聊了三个月，它还记得你第一个月说过的事。这就是"越用越懂你"的底层支撑。

这次攻击的规模：135,000+台设备受影响，ClawHub约20%的Skills被确认为恶意。如果你随机安装5个Skill，大概率至少有1个有问题。[^1^]

这件事给所有OpenClaw用户敲响了警钟：Skill本质上就是**受信任代码**——一旦安装，它就拥有和你的OpenClaw实例相同的权限，没有沙箱隔离。安装前务必审查源码，优先使用社区的精选列表，定期检查SOUL.md和MEMORY.md是否被异常修改。[^1^]

## 3.5 为什么OpenClaw如此成功

好，讲了这么多技术细节，我们来回答一个更重要的问题：OpenClaw凭什么火成这样？

我认为有四个核心原因，每一个都精准命中了当下用户的真实需求。

### 原因一：数据主权——完全自托管，数据不出境

这是**央企和金融机构最核心的诉求**。OpenClaw装在你自己的服务器上，所有数据——聊天记录、邮件内容、日程安排、生成的文件——全部保存在本地。不经过任何第三方平台，不发送到国外。[^1^]

大家想想，中国移动作为央企，对数据合规的要求是极高的。OpenClaw的"自托管"特性意味着你可以完全掌控数据的流向——Agent帮你处理的所有信息都在你自己的机房或云服务器上，不存在数据出境的风险。Loopback-First的默认安全设计也强化了这一点：不主动暴露外网端口，想要远程访问需要通过Tailscale等安全通道。[^1^]

### 原因二：模型自由——不被任何厂商绑定

OpenClaw支持十余家模型提供商，从国际顶尖的Claude、GPT、Gemini，到国产的DeepSeek、GLM、通义千问、豆包、Kimi、文心、MiniMax，再到完全免费的本地模型Ollama和LM Studio。[^1^]

你可以通过一个配置文件随时切换主力模型，还能设置Fallback备选链——主模型不可用时自动降级到更便宜的模型。这种"模型自由"让你永远不会被单一厂商锁定。国产模型表现越来越好了：GLM-5的代码能力接近Claude Opus 4.5水平，DeepSeek-V3.2.2的推理能力也足以应对大多数日常任务。[^1^]

### 原因三：渠道覆盖——全面贴合国内企业办公习惯

这是OpenClaw在中国市场爆发的关键。国内四大主流办公IM——飞书、钉钉、企业微信、QQ——全部支持。尤其是飞书，自OpenClaw 2026.2版本起获得**原生内置支持**，OpenClaw官方和飞书都提供了详细的集成文档。[^1^]

还有openclaw-china这个统一插件，三步完成国内五大平台的接入。国内社区"云养虾"用户已超10万，深圳龙岗AI局甚至发布了OpenClaw使用支持措施的征求意见稿——一个开源项目引发地方政府的政策关注，这在国内并不多见。[^1^]

### 原因四：成本可控——国产模型价格仅为国际模型的1/20

来看一组直观的对比数据：[^1^]

| 模型 | 输入价格/百万tokens | 输出价格/百万tokens |
|------|-------------------|-------------------|
| Claude Sonnet 4.6 | **$3.00** | $15.00 |
| GPT-5.4 | $2.50 | $15.00 |
| Gemini 3 Pro | $2.00 | $12.00 |
| **DeepSeek-V3.2** | **$0.14** | **$0.28** |
| **GLM-5** | **$0.80** | **$2.56** |
| **豆包 Seed 2.0 Pro** | **$0.47** | **$2.37** |

DeepSeek的输入价格仅为Claude Sonnet的**约1/21**。配合Fallback链策略——简单任务走便宜的DeepSeek，复杂任务才用Claude——整体API成本可以降低**80%-95%**。再加上免费的本地模型（Ollama）和国产云厂商的Coding Plan包月套餐（首月低至7.9元），一套完整的Agent系统月度成本可以控制在百元以内。[^1^]

当然，成本低的前提是做好控制。OpenClaw的多轮思考和多工具调用可能消耗传统聊天几十甚至上百倍的token，社区中"一觉醒来收到$1,100 API账单"的恐怖故事确实存在。务必设置每日预算上限，配置好Fallback降级链。[^1^]

【展开讲】OpenClaw的成功还有一个软实力因素——"养虾"文化。因为吉祥物是龙虾，中文社区把运行OpenClaw称为"养虾"，用户自称"养虾人"。"你养龙虾了吗？"成了AI圈的问候语。这种有趣的文化标签让一个技术项目有了社交货币的属性，2026年3月6日深圳腾讯云总部近千人排队体验OpenClaw安装——一个开源项目能引发这样的线下热度，技术能力之外，文化现象级的传播力也功不可没。[^1^]

## 3D.1 Unix哲学：小工具、可组合、文本流

OpenClaw的技术选择背后有一套清晰的设计哲学，直接继承自Unix：**小工具、可组合、文本流。**

创始人Peter Steinberger的观点很明确："CLI才是智能体连接世界的终极接口。"不需要为每个服务写一个专门集成，Agent只要能运行命令行，就能操作一切。[^1^]

在这个理念指导下，OpenClaw的核心工具被刻意精简到**只有4个**：[^1^]

| 工具 | 用途 |
|------|------|
| Read | 读取文件 |
| Write | 写入文件 |
| Edit | 编辑文件 |
| Bash | 执行命令 |

就这四个。不是功能缺失，而是刻意为之——更少的工具意味着更短的system prompt、更少的token消耗、更快的响应。这四个工具足以覆盖几乎所有操作系统级别的任务，再加上Skills生态的扩展，能力可以无限延伸。

## 3D.2 对MCP的态度

MCP（Model Context Protocol）是Anthropic推出的工具协议标准，几乎整个行业都在集成。但OpenClaw的态度很有意思——**创始人直言"MCP不能scale"**。[^1^]

Peter的原话是："我的前提是MCP是垃圾，不能scale。你知道什么能scale？CLI。Unix。"[^1^]

OpenClaw的替代方案是：Agent通过Bash工具直接调用CLI程序，不需要中间协议层。对于确实需要MCP的场景，它提供了一个`mcporter`技能来做桥接，但核心设计理念是"强制Agent自己扩展能力，而非消费预构建的MCP工具集"。

这个观点对不对？有争议。但OpenClaw的成绩说明，至少对于它的目标场景——自托管个人AI助手——这条路线是走得通的。

## 3D.3 OpenClaw的局限性

讲完了优点和设计哲学，我们必须客观地说说OpenClaw的问题。给大家一个全面的认知。

### 资源占用不轻

OpenClaw的代码规模约**43万行TypeScript**，运行时内存占用约**1GB**，启动时间3-5秒。这并不"轻量"。但作为一个24/7运行的个人AI助手，在现代硬件上完全在可接受范围内。[^1^]

### Skills质量参差不齐

ClawHub上注册的13,729个Skills中，**超过50%被社区判定为垃圾、重复或低质量**。更危险的是约20%在高峰期被确认为恶意。生态质量问题需要每一个用户自己把关。[^1^]

### 安全事件历史

不到4个月的历史里，OpenClaw经历了至少两起重大安全事件：[^1^]

**CVE-2026-25253**——CVSS评分8.8/10的高危远程代码执行漏洞。攻击者可以伪造WebSocket origin header连接到暴露的Gateway，在实例上执行任意代码。该漏洞已在v2026.3.2版本中修复。

**ClawHavoc供应链攻击**——前面已经详细讲过，135,000+台设备受影响，Agent的长期记忆文件被篡改。

### 创始人坦承的风险

值得一提的是，Peter Steinberger对安全问题保持了罕见的坦诚。他的原话：[^1^]

> "This is all vibe code. Prompt injection hasn't been solved. There are absolute risks."

（这全是vibe code。Prompt injection没有被解决。存在绝对风险。）

这种坦诚值得尊重，但也意味着：**如果你要在生产环境中使用OpenClaw，安全防护必须由你自己负责。** OpenClaw提供了基础的安全机制，但远谈不上"企业级安全"。DM配对保护、群组沙箱模式、工具访问控制、Gateway强制认证——这些都需要你主动配置和维护。[^1^]
