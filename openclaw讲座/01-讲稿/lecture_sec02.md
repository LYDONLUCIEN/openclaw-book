## 2.1 ReAct范式：Agent的"思考-行动"循环

### Agent到底是怎么工作的

大家想想，我们人类平时遇到一个复杂问题会怎么做？

比如领导让你"分析一下上个月的客户投诉数据"。你不会立刻开始动手，而是先在脑子里过一遍：我要找谁要数据？需要什么口径？分析哪些维度？——这是**思考**。然后你去找IT部门拉数据——这是**行动**。看到数据之后你发现字段不全，需要再补充几个维度——这是**观察结果、调整方案**。然后再想、再做，直到报告完成。

ReAct就是这个逻辑。[^26^]

它的全称是 Reasoning + Acting，由普林斯顿大学和谷歌研究院在2022年提出，2023年发表在顶会ICLR上。[^41^] 核心思想特别简单——**让AI像人一样，边想边做，做错了就调整，直到把事情做完**。

### 传统AI vs Agent的本质区别

传统AI是什么模式？你问一句，它答一句，对话结束。就像你去餐厅问服务员"这道菜辣吗"，他回答你"微辣"，然后就没有然后了。

Agent是什么模式？你说"帮我准备一份客户投诉分析报告"，它不会立刻给你生成一段文字就完事。它会经历一个完整的循环：[^30^]

> 用户输入 → **思考**"需要哪些数据源" → **选择工具**调用数据库 → **执行**查询 → **观察**"数据缺了投诉分类字段" → **重新思考**需要补充维度 → **再次行动** → … → **判断完成** → 返回完整报告

每一步都建立在真实数据的基础上，而不是凭空编造。这也是为什么Agent能大幅减少"胡说八道"的概率——它每一步都在跟真实世界交互，用事实来校正自己的判断。[^32^]

更重要的是，这个循环可以持续很多轮。传统AI是一次性回答，Agent是**持续迭代直到完成**。[^31^] 就像刚才说的那个故障排查的例子，Agent可能需要五到六轮的思考→行动→观察，才能把根因找到、报告写好、邮件发出去——但全程不需要你一步步指导，它自己就知道下一步该干什么。

ReAct的价值，说白了就是把人类解决问题的自然认知过程，变成了Agent的标准工作方式。[^31^]

## 2.2 MCP协议：Agent连接世界的"通用插头"

### 一个让人头疼的问题

Agent要做事，就得连上外部系统——查数据库、读文件、发邮件、调用API。但问题来了：市面上的AI框架千千万，外部工具万万千，以前每个框架连每个工具都要单独写适配代码。这叫"M×N困境"——N个框架 × M个工具 = N×M个集成。[^50^] 工作量是指数级爆炸的。

2024年11月，Anthropic发布了一个叫MCP的开放标准协议，全名Model Context Protocol，模型上下文协议。[^42^] 业内直接把它比作**AI世界的USB-C接口**。

### USB-C的类比

大家手上的手机、笔记本，以前充电口各不相同——圆的、扁的、宽的，出个门带一堆线。USB-C出来以后，一根线走天下，充电、传数据、接显示器都用它。

MCP干的就是同样的事。以前Agent框架连数据库要写一套代码，连浏览器要写一套，连文件系统又要写一套。有了MCP，工具提供方按MCP标准暴露一次能力，任何兼容MCP的Agent框架都能直接使用。[^49^] 集成量从 N×M 降到了 N+M。

这意味着什么？意味着Agent可以通过MCP访问数据库、浏览器、文件系统、邮件系统——**万物互联**。[^48^]

MCP的行业影响力非常大。发布半年左右，MCP Registry上就收录了接近2000个Server，OpenAI、Google、Microsoft、AWS都宣布接入，国内阿里云、腾讯云、字节飞书也相继集成。[^63^] 2025年底，连Linux基金会都专门成立了Agentic AI Foundation来治理MCP协议。[^160^] 可以预见，未来企业的核心业务系统只要接入了MCP，Agent就能像插U盘一样即插即用。

## 2.3 Skills系统：Agent的"专业技能证书"

### 什么是Skill

有了ReAct的"思考-行动"能力和MCP的"连接能力"，Agent理论上能做任何事了——但它真的知道怎么做吗？

打个比方，给了你一台连了全网的高端电脑，让你做一份"5G基站能耗优化方案"，你也不知道从哪下手。你需要的是方法论——先查什么指标、再比什么数据、用什么模型分析、按什么格式输出。

**Skill就是这个"方法论"。** 它是教Agent按固定流程做事的说明书，一次写好，反复使用。[^52^]

Skill由Anthropic在2025年底发布为开放标准，文件名叫SKILL.md，采用Markdown格式。截至2026年，已有**30+主流平台采用**，包括Claude Code、VS Code、GitHub Copilot、Cursor等。[^51^]

一个标准的SKILL.md长什么样？三段式结构：

| 部分 | 作用 |
|------|------|
| 元数据（名称+描述+触发条件） | 告诉Agent"我是谁、在什么情况下该用我" |
| 工作流步骤 | 告诉Agent"具体怎么做，第一步干什么、第二步干什么" |
| 约束规则 | 告诉Agent"永远不要做什么、必须遵守什么底线" |

### Skill示例1：日报生成Skill

这是一个贴近大家日常的例子。当用户说"帮我写日报"时，Agent加载这个Skill，按以下流程执行：[^52^]

> 询问用户今日完成的工作事项 → 按项目/分类整理 → 标注每项的完成状态（已完成/进行中/阻塞） → 按Markdown格式排版 → 自动保存到指定位置

这个过程里，Skill规定了"先问再写"（不会自己编造工作内容）、"按项目分类"（不会一团乱麻）、"标注状态"（让日报有信息量）、"Markdown格式"（便于后续处理）——每一步都有章可循。

### Skill示例2：代码审查Skill

再举一个技术场景的例子：[^123^]

> 读取Pull Request中的所有变更文件 → 逐行分析安全风险（如OWASP Top 10漏洞） → 检查代码风格一致性 → 验证测试覆盖率 → 按结构化格式输出审查报告

约束规则写得很清楚：**永远不要**未经完整检查就自动批准PR；**永远不要**用主观判断覆盖安全发现；**必须**为每个问题标注具体行号。

### Skill与Tool的区别

很多人分不清Skill和Tool，我用一个比喻：[^44^]

**Skill是食谱，Tool是厨具。** 食谱告诉你"先处理食材、在特定温度下烹饪、按固定比例调味"——但光有一本食谱你做不出菜，你还需要刀、锅、炉灶（Tool）来真正动手。反过来说，给你全套厨具但没有食谱，你也可能把菜炒糊。

Agent的Tool是 "能做什么"——查天气、读文件、发邮件。Skill是"该怎么做"——遇到什么情况按什么流程做。两者缺一不可。

## 2.4 多Agent协作：复杂任务的"团队作战"

### 一个人干不完的活，交给一个团队

前面讲的都是一个Agent在干活。但现实中，很多任务一个人搞不定——比如写一份高质量的行业分析报告，既要搜集资料、又要分析数据、又要动笔写、还要审校质量。让一个Agent全包了，结果就是"样样通、样样松"。

多Agent协作的思路很简单：**主Agent当项目经理，负责拆解任务和汇总结果；多个子Agent各司其职，各自负责专业领域。** [^43^]

### 层级模式：项目经理制

具体怎么运作？以"写一份行业分析报告"为例：[^55^]

> 主Agent（项目经理）接到任务 → 拆解为四个子任务 → **研究员Agent**负责搜集行业资料和数据 → **分析师Agent**负责整理数据、做对比分析 → **写手Agent**负责生成报告正文 → **审核Agent**检查质量和事实准确性 → 主Agent汇总各子Agent输出 → 自我纠错、合并冲突 → 输出最终报告

这种模式的核心优势是专业化分工和并行效率。研究员不用分心想着怎么排版，写手不用操心数据从哪来——就像真正的团队一样，每个人专注自己擅长的部分。[^59^]

目前，Agent之间的通信也有了行业标准。Google推出的A2A协议（Agent-to-Agent Protocol）专门解决不同Agent之间的协作通信问题，与MCP形成互补——MCP解决"Agent连工具"，A2A解决"Agent连Agent"。[^63^]

对于中国移动这样的大型企业，多Agent协作的意义特别明显。我们的业务场景复杂度很高——一份省级经营分析报告可能需要数据提取、指标计算、趋势分析、对标比较、文案撰写等多个环节，每个环节交给专业的Agent来做，整体质量和效率都会显著提升。

> 【展开讲】Skill的核心价值与设计原则

### 2D.1 为什么Skill比单纯Prompt更有效

有人可能会问：我直接在对话框里写一段详细的指令，不就是Skill吗？差别大了。

第一个差距是**成本**。研究表明，如果把90个工具的定义一次性塞进Agent的上下文，要消耗**50,000+ tokens**。而采用Skill的渐进式加载——先只显示名称和描述，需要时再展开完整内容——初始开销只有约**2,700 tokens**，节省了**94%**。[^134^]

第二个差距更关键——**效果**。LangChain实验室做过一个对比实验：同样的任务，不用Skill时通过率只有**25%**，加载对应的Skill后通过率提升到**95%**——提升了近4倍。[^52^] 这不是换了一个更贵的模型做到的，仅仅是给Agent配了一份好的操作手册。这说明在正确的场景下，**好的Skill设计比更大的模型更有价值**。

### 2D.2 好的Skill设计原则

Skill的设计是一门工程学问。三个核心原则：

**第一，单一职责。** 一个Skill只做一件事。不要写出一个"万能Skill"，又管代码审查、又管部署上线、又管Bug修复。职责越单一，Agent触发越精准，行为越可控。[^52^]

**第二，约束先行。** 写Skill的时候，先想"这个Skill绝对不能做什么"，把约束写在最前面。因为Agent天生倾向于"帮助用户"，如果没有明确禁止，它可能会"创造性地"绕过你没有规定的步骤。没有约束的Skill是危险的Skill。[^52^]

**第三，渐进式披露。** Agent先只看到Skill的名称和一句话描述（约30-50 tokens），判断跟自己当前任务相关后，才加载完整的工作流程。这就像图书馆的检索系统——你不会先把所有书都搬到桌上再找，而是先看书目，觉得有用才去取。[^134^]

这三个原则配合起来，Skill才能在实际生产中稳定、可靠、可维护地运行。对于中国移动来说，这意味着我们能把一线的业务专家经验——比如客服SOP、运维流程、审批规范——固化成标准化的Skill，让Agent按照经过验证的最佳实践来执行，而不是每次都"即兴发挥"。
