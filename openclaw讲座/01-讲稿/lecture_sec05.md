# 5. Agent的能力全景：能做什么、不能做什么

前面四章，我们从概念到架构、从对比到选型，一步步把Agent的"底细"摸清楚了。但到这里，大家心里可能还有一个最实际的问题——

**"这东西到底能帮我把工作做到什么程度？"**

这一章，我们就来画一张Agent的"能力地图"，把擅长和不擅长的边界标清楚，再教大家几招进阶用法，最后用一个完整的网络运维周报场景，带大家从头到尾看一遍Agent是怎么干活的。

---

## 5.1 Agent的"能力地图"

Agent的能力不是"什么都会"，也不是"什么都懵"，它有非常清晰的擅长领域和明确的能力边界。理解这个边界，是用好Agent的第一步。

### 5.1.1 擅长的事情：五大核心能力

**① 信息处理——阅读/摘要/整理海量文档**

这是Agent最擅长的基本功。一份百页的技术文档，Agent能在几十秒内提取关键信息、生成摘要、回答针对文档的具体问题。OpenClaw内置的Summarize技能就是干这个的——视频、网页、邮件内容都能自动摘要。[^1^] 更实用的是，Agent可以同时处理多份文档，交叉比对、提炼共性、发现矛盾。比如把三大运营商的财报一起丢给Agent，它能快速拉出各自的5G用户数、ARPU值、资本开支，做成一张对比表。

【展开讲】这里的核心技术是RAG（检索增强生成）。Agent把文档切分成小块，建立向量索引，查询时先检索最相关的片段，再基于这些片段生成回答。这种方式既克服了大模型上下文长度的限制，又能处理百页甚至千页级别的材料。[^2^]

**② 内容生成——报告/PPT/邮件/代码**

Agent不是简单的"文字生成器"。给它一组数据和明确的格式要求，它能输出结构完整的Markdown报告、格式规范的邮件草稿、甚至可以直接运行的Python脚本。在金融投研领域，已有券商实现"每日A股公告自动抓取汇总"，从原来的2小时手工整理压缩到30秒自动完成。[^3^] 

但大家要有一个正确预期——Agent生成的内容，**框架和逻辑通常很扎实，细节和语气需要人工润色**。把它当成一个"初稿写手"，而不是"终审编辑"。

**③ 系统操作——浏览器自动化/命令执行/文件处理**

这是Agent区别于传统AI助手的核心能力。OpenClaw的Browser技能基于Chrome DevTools Protocol，可以自动登录网页后台、填写表单、截图、导出PDF。[^1^] 一条指令"打开CSDN，搜索10篇OpenClaw文章，每篇总结一下，保存为CSV"，Agent就能完整执行搜索→访问→总结→保存的全流程。[^4^]

在中国移动的实际工作场景中，这意味着Agent可以帮你：自动登录运维监控平台抓取告警数据、在OA系统填写标准化表单、从多个内部系统导出数据并整合。

**④ 数据分析——提取/清洗/可视化**

Agent能执行完整的数据处理流水线。读取原始日志→提取关键字段→清洗异常值→生成统计图表→输出分析结论。OpenClaw内置55个技能，配合ClawHub技能市场上超过13,000个社区技能，数据分析相关的技能覆盖Excel处理、Python数据分析、图表生成等多个方向。[^1^]

**⑤ 流程自动化——定时任务/条件触发/跨系统协作**

OpenClaw的HEARTBEAT.md可以定义定时任务和主动行为，比如每30分钟检查一次任务状态、每天早8点推送运维日报。[^5^] 更强大的是跨系统协作——Agent可以同时连接运维监控系统、邮件系统、飞书/钉钉，实现"发现异常→自动排查→发送告警→创建工单"的全链路自动化。

### 5.1.2 不擅长的事情：四条能力红线

说了擅长的，更要诚实说**不擅长的**。这四条红线，大家务必记住：

**① 高精确度数学计算——可能"幻觉"出错**

大模型本质是"文字预测引擎"，不是计算器。两个六位数相乘，Agent大概率会算错。因为它不是在执行算术运算，而是在"猜测"最可能的数字组合。需要精确计算的场景，一定要借助外部工具（如Python计算库）来兜底。[^6^]

**② 物理世界操作——无法触摸实物**

Agent能控制浏览器、操作文件、发送消息，但它不能帮你搬服务器、插网线、按电源按钮。它能生成一份"机房巡检清单"，但走进机房检查温度、观察指示灯状态，还是需要人。

**③ 创意审美判断——生成内容需人工确认**

Agent可以生成PPT框架、撰写宣传文案，但审美层面的判断——这个配色是否协调、这句文案是否符合品牌调性、这个排版是否好看——目前还需要人的把关。Agent没有"审美直觉"，它只有"训练数据中的统计规律"。

**④ 需要人类授权的场景——财务审批、人事决策**

涉及钱、权、人的决策，Agent只能辅助分析，不能代替拍板。Agent可以帮你整理报销票据、生成审批流程的报告摘要，但最终的签字确认必须是人。这是能力边界，更是安全底线。

> 小结一下：Agent是"数字世界的操作员"——在信息、数据和系统之间游刃有余；但它不是"全能超人"——精确计算、物理操作、审美判断、授权决策，这四件事需要人的参与。

## 5.2 进阶使用技巧

掌握了基本能力之后，我们来看看四个让Agent"更上一层楼"的进阶技巧。这些技巧组合起来，就是把Agent从"工具"变成"团队"的关键。

### 5.2.1 Skills组合：多个Skill串联完成复杂任务

单个Skill解决单个问题，但把多个Skill串起来，就能完成复杂的工作流。一个典型的组合是：

**搜索（Web Search）→ 研究（Deep Research）→ 生成（Image Gen / Document）→ 发送（Gmail / 飞书）**

自媒体博主的真实案例：用`brave-search`搜索热点→`deep-research`深度分析→AI撰写文章→`fal-ai`生成配图→`notion`保存草稿，每天产出从3篇提升到5篇，时间从6小时压缩到2小时。[^4^]

在中国移动的场景里，这个组合可以变成：**运维监控Skill抓告警 → 搜索Skill查解决方案 → 报告Skill生成故障分析 → 飞书Skill推送给值班群**。

### 5.2.2 SubAgent分工：让不同"专才Agent"各司其职

复杂任务由一个Agent包办，容易"顾此失彼"。更好的做法是像组建项目团队一样，让不同的SubAgent各司其职。[^7^]

| 角色 | 职责 | 中国移动场景示例 |
|------|------|-----------------|
| 研究Agent | 负责信息收集和数据获取 | 从多个运维系统拉取性能指标 |
| 分析Agent | 负责数据整理和趋势分析 | 识别异常波动、定位根因 |
| 写作Agent | 负责生成报告和可视化 | 撰写周报正文、制作趋势图 |
| 审核Agent | 负责质量检查和合规审查 | 确认数据完整性、检查敏感信息 |

主Agent扮演"项目经理"的角色——接到任务后分解、分配给各个SubAgent、收集结果、最终汇总输出。[^7^]

### 5.2.3 自我纠错机制：Agent发现不对→自动重试→修正→输出

这是Agent与传统自动化工具**最本质的区别**。传统脚本跑错了就报错退出，Agent能"自己发现问题、自己想办法"。

背后的核心机制叫Reflexion框架——Agent在任务失败后进行口头反思，把反思结论存入记忆缓冲区，下次尝试时参考。[^8^] 比如代码调试中，上次失败后反思"变量count在调用前未初始化"，下次直接规避同类错误。再比如数据分析时，Agent发现某个指标值为空，它会自动回溯检查数据源、尝试其他获取途径、或者在报告中标注"数据缺失"。

这种"行动→反思→优化"的循环，让Agent在不确定的环境下也能可靠工作，而且不需要人工标注训练数据，完全靠自然语言形式的自我反馈来改进。[^8^]

### 5.2.4 结果沉淀：每次执行的学习成果自动保存为Skill或记忆

OpenClaw的四层记忆系统（SOUL/TOOLS/USER/Session）中，USER层的MEMORY.md会持久化保存Agent在对话中积累的用户偏好和决策记录。[^5^] 这意味着——

你第一次让Agent写周报，需要反复调整格式；第二次，Agent已经记得"你习惯把故障分析放在第3节、喜欢用柱状图而不是折线图"；到第十次，它可能主动说"我发现你每周都手动添加基站退服率这个数据，我把它固化为Skill吧，以后自动抓取"。

这就是**效率的复利增长**——每次使用都在让Agent变得更懂你。

## 5.3 完整场景穿透：从用户指令到最终结果

前面讲的是"地图"和"技巧"，现在我们来走一次**完整的实战流程**。这个场景贴近中国移动网络运维的真实工作——**"帮我生成本月的网络运维周报"**。

大家跟上节奏，看Agent是怎么把这10步走完的。

### 5.3.1 场景设定

假设你是地市公司网络运维主管，每周五需要提交一份网络运维周报，内容涵盖：基站运行状态、告警统计、故障处理情况、性能指标趋势、下周工作计划。数据来源分散在多个系统中——运维监控平台、工单系统、性能管理平台。

以前，你需要逐个系统登录、导出数据、手动整理、撰写报告，**整个过程2-3小时**。现在，你把这活儿交给Agent。

### 5.3.2 全流程模拟：10步穿透

**第①步：用户输入指令**

你在飞书对Agent说："帮我生成本月的网络运维周报，要包含基站状态、告警统计、故障处理、性能趋势和下周一计划，格式跟上周一样。"

**第②步：主Agent理解目标并分解任务**

主Agent（Orchestrator）开始工作。它先从MEMORY.md中读取"上周周报"的记录，确认格式模板；然后把任务拆解为5个子任务：数据采集×3、数据分析、报告生成。[^7^]

**第③步：调度数据采集Agent从多个系统获取数据**

主Agent同时启动3个数据采集SubAgent：

- 运维监控Agent → 登录性能管理平台 → 抓取本月基站在线率、退服率、告警数量
- 工单系统Agent → 访问工单系统 → 提取故障工单列表、处理时长、闭环率
- 性能分析Agent → 访问性能管理平台 → 获取KPI趋势数据（流量、接通率、掉话率）

这里每个SubAgent都在执行ReAct循环——思考→行动→观察，直到成功拿到数据。[^9^]

**第④步：数据分析Agent整理分析**

数据汇聚后，分析Agent开始干活：

- 清洗异常数据（比如某天的0值是因为系统维护，不是真实故障）
- 计算同比/环比变化（本月基站在线率99.2%，比上月提升0.3%）
- 识别Top 3频发告警类型和趋势
- 生成趋势分析结论

**第⑤步：报告生成Agent生成Markdown报告**

写作Agent按照上周模板，将分析结果组织成结构化报告：

```
# 网络运维周报（2026年6月第3周）
## 一、基站运行状态
## 二、告警统计分析
## 三、故障处理情况
## 四、性能指标趋势
## 五、下周工作计划
```

每个章节自动插入数据表格和趋势图引用。

**第⑥步：格式转换Agent转为Word/PPT**

报告生成后，格式转换Agent将Markdown转为Word文档和PPT两种格式，方便不同场景使用。

**第⑦步：审核Agent检查内容完整性**

审核Agent登场，逐项核对：

- 五节内容是否齐全？→ 齐全 ✓
- 数据是否与上周格式一致？→ 一致 ✓
- 关键指标是否有异常缺失？→ 发现"基站退服率"字段为空 ✗

**第⑧步：自我纠错——发现数据缺失→补充获取**

审核Agent发现"基站退服率"数据缺失，主Agent触发纠错流程：

1. 回溯检查：分析Agent确认退服率字段确实为空
2. 诊断原因：运维监控Agent发现性能管理平台该指标更名为"基站不可用率"
3. 补充获取：按新指标名称重新抓取数据
4. 重新生成：写作Agent更新报告，审核Agent再次确认 ✓

这一步就是Agent自我纠错机制的真实体现——不是"报错退出"，而是"发现问题→诊断原因→自动修复"。[^8^]

**第⑨步：最终结果交付用户**

主Agent将最终的周报（Word + PPT两种格式）发送给你，附带一句话说明：

> "周报已生成。本周基站在线率99.2%（环比+0.3%），共处理故障工单47件（平均处理时长3.2小时）。注意：性能管理平台'基站退服率'已更名为'基站不可用率'，已自动适配。请确认后提交。"

**第⑩步：沉淀为"周报生成Skill"供下次直接使用**

最后一步最关键——Agent询问你："是否将本次周报的生成流程保存为Skill？下次你只需说'生成本周周报'，我就能自动执行全部步骤。"

你确认后，Agent将完整的流程（数据源、格式模板、分析逻辑、输出格式）写入一个名为`weekly-report-generator`的Skill，保存在工作区的skills目录下。[^10^] 下次使用时，**整条10步流程压缩成一句话**。

### 5.3.3 可视化流程总结

把这10步串起来，数据流转是这样的：

```
用户指令 → 主Agent分解 → 多路数据采集（监控平台|工单系统|性能平台）
    ↓
数据汇聚 → 分析Agent清洗/计算 → 写作Agent生成报告
    ↓
格式转换 → 审核Agent质检 → 【纠错循环】→ 终稿交付
    ↓
用户确认 → 沉淀为Skill → 下次一句话调用
```

整个过程的核心数据：

| 指标 | 手工方式 | Agent方式 |
|------|---------|----------|
| 时间消耗 | 2-3小时 | 5-10分钟 |
| 数据源数量 | 逐个手动导出 | 3个系统并行抓取 |
| 人工干预次数 | 全程操作 | 仅需最终确认 |
| 可复用性 | 每次重复劳动 | 首次后一键执行 |

## 5D.1 培养Agent的3个小技巧

Agent不是"开箱即用"就完美的，它需要你的培养。三个小技巧，帮你快速建立高效的协作关系。

### 5D.1.1 从简单任务开始：先让Agent做单一明确的事

刚开始不要直接丢一个复杂的10步流程。先让Agent做单一、明确的事——"帮我把这个Excel里的重复行去掉"、"把这段文字翻译成英文"。等它把简单的事情做稳了，再逐步增加任务的复杂度。这跟带新员工的逻辑是一样的，先给简单的活，建立信任，再逐步加码。

### 5D.1.2 及时反馈修正：Agent做错时及时纠正，它会记住你的偏好

Agent的输出不符合预期时，不要默默接受或重新自己做。明确告诉它哪里不对、应该怎么改。OpenClaw的记忆系统会把你的反馈写入MEMORY.md，下次遇到类似场景自动调整。[^5^] 比如你说"周报里故障分析部分要放在性能趋势前面"，Agent记住后，以后每次都会按这个顺序排。

### 5D.1.3 建立个人Skills库：把重复性工作流程固化为Skill

凡是每周重复做的事——周报、月报、数据汇总、邮件通知——都值得固化成Skill。一个Skill的最小单位就是一个目录加一个SKILL.md文件。[^10^] 你的Skills库越丰富，Agent的"经验值"越高，效率提升是复利式的。

## 5D.2 成本控制要点

Agent的能力强大，但用错了模型、不设消费限额，可能一觉醒来收到天价账单。社区里"一晚$1,100 API账单"的恐怖故事可不是个例。[^1^] 两个关键要点，帮你在享受能力的同时守住钱包。

### 5D.2.1 模型费用是大头：合理选择国产模型可节省90%+

服务器成本已经非常低了——阿里云9.9元/月、腾讯云17元/月都有。[^1^] 真正的持续开销是模型调用费用。来看一组对比数据：

| 模型 | 输入价格/百万token | 相对成本 |
|------|-------------------|---------|
| Claude Sonnet 4.6 | **$3.00** | 100%（基准） |
| GLM-5 | **$0.80** | 约27% |
| DeepSeek-V3.2 | **$0.14** | 约5% |
| Ollama本地模型 | **免费** | 0%（仅电费） |

同样的任务，用DeepSeek-V3.2替代Claude Sonnet，成本直接降到5%。日常任务用国产模型，复杂任务再用国际顶尖模型，这是目前性价比最优的策略。[^1^]

### 5D.2.2 设置消费限额：避免意外高额账单

OpenClaw支持在配置中设置预算上限：[^1^]

```json
{
  "agents": {
    "defaults": {
      "budget": {
        "maxTokensPerDay": 500000,
        "maxCostPerDay": 5.00
      }
    }
  }
}
```

强烈建议每个人都设置日预算上限。哪怕你不差钱，一个每日5美元的上限也能在Agent意外进入循环推理时保护你的钱包。这是花5秒钟、省几千块的必要操作。
