## S02 不可能三角 + 成本视角

### PPT展示内容（一屏完整）

```
标题：所有系统都逃不过的三个特性

三角图（三个顶点）：
  ┌─────────┐
  │ 确定性   │ ← 输出靠谱 → 确认成本
  │靠不靠谱  │   （人的智能消耗在审判、决策）
  └────┬────┘
       │
 ┌─────┴──────┐
 │便利性      │完备性│
 │省不省心    │能干多少活，过程用不用人操心│
 │ → 操作成本  │ → 开发成本│
 │（调度执行） │（开发建设）│
 └────────────┘

底部金句：
三个都想做好？→ 需要持续投入。成本不会消失，只会转移。
```

### 整体目标
建立全场核心框架：三个特性不可得兼，每一种特性不足对应的都是一种成本。让小白也能用大白话理解。

### 串讲词
> "所有软件系统，不管多复杂，本质上都在追求三个好特性：确定性——靠不靠谱；完备性——能干多少活；便利性——用起来省不省心。"
>
> "但这三个特性不可能同时做到满分。为什么？因为每一项背后都是人的成本——操作成本、开发成本、确认成本。你想减少操作，就要多开发；你想减少确认，就要多操作。"
>
> "记住这个三角，后面所有内容都会回到这个框架。"

### 推荐优化方向
- 三角图可交互：点击每个顶点展开"大白话"解释和成本对应关系
- 大白话标签（靠不靠谱/能干多少活/省不省心）用较小字号显示在特性名下方
- 展开后显示更多详细说明，但初始状态必须一屏完整

### 复用
现有 `Slide02_Chaos`（三个特性齿轮卡片），改造为三角图+成本标签形式。

```
│完备性 10  │ │完备性 20  │ │完备性 35  │ │完备性 90  │
│便利性 85  │ │便利性 75  │ │便利性 80  │ │便利性 60  │
├──────────┤ ├──────────┤ ├──────────┤ ├──────────┤

```

### 整体目标
用数值化对比让小白一目了然：每种技术路线的"特性加点"不同，对应的成本也不同。引出大模型的位置——完备性最高但确定性最低。

### 串讲词
> "我们来看四种常见的技术路线。传统代码，什么都写得出来，但开发成本最高，完备性最低——你得一行行写。工作流好一点，流程标准化了，但灵活性差。机器学习需要大量数据喂养。"
>
> "大模型就不一样了——它什么都能聊，完备性最高。但是，它一本正经胡说八道的概率也是最高的。确定性只有20分。"
>
> "那我们能不能给大模型穿装备，把确定性补回来？这就是龙虾做的事情。"

### 推荐优化方向
- 保留现有的属性雷达图/柱状图可视化
- 每个卡片底部加一句"大白话"描述（如大模型："什么都懂，但不靠谱"）
- 点击卡片展开时显示对应的成本分析和内部系统举例
- **[待补充]**：每个路线可标注对应的中国移动内部系统名称

### 复用
现有 `Slide03_Ecosystem`（四种职业图鉴卡片+属性条），添加成本标签。

## S07 v1.0 绿装 — 人工教学

### PPT展示内容（一屏完整）

```
标题：v1.0 绿装 — 人工教学

┌────────────────┐  ┌────────────────┐
│   提示词工程    │  │  Function Call  │
│   Prompt Eng   │  │                │
├────────────────┤  ├────────────────┤
│ 解决：输入模糊  │  │ 解决：处理不确定 │
│ · 角色设定     │  │ · 明确工具接口   │
│ · 任务描述     │  │ · 结构化输出     │
│ · 格式要求     │  │ · 可调用的函数   │
│ · 示例给出     │  │                │
│ · 约束条件     │  │                │
└────────────────┘  └────────────────┘

特性变化：确定性↑ 完备性↑ 便利性·
成本变化：操作成本↑（要写好Prompt） 开发成本· 确认成本↓

相当于：手把手教应届生做事
```

### 整体目标
展示最早的解决方案——人工通过Prompt和Function Call来弥补不确定性。强调这是"操作成本"的投入。

### 串讲词
> "最早的方案就是人工教学。你把任务写得清清楚楚——角色是什么、做什么、格式怎样、给个例子——这就是提示词工程。再给他明确的工具接口，让他调用函数而不是自由发挥，这就是 Function Call。"
>
> "确定性上来了，完备性也上来了。但便利性没变——因为每次你都要花大量时间写好 Prompt。这就是操作成本的投入。"
>
> "相当于手把手教应届生，你说一步他做一步。"

### 推荐优化方向
- Prompt 五要素可以用简化的代码块展示（现有Slide07已有）
- Function Call 用简单的 JSON 片段展示接口定义
- 展开后可显示真实 Prompt 示例和 Function Call JSON
- 底部"特性变化"条用绿装配色

### 复用
现有 `Slide07_ActionProblem`（Tab切换：Prompt / Function Call），添加成本标签和特性变化指示。

## S09 v3.0 紫装 — 自主调度

```
标题：v3.0 紫装 — 自主调度

Tab: [ ReAct ]  [ MCP ]

ReAct 循环：
┌─────┐    ┌─────┐    ┌─────┐
│思考  │ →  │行动  │ →  │观察  │
│Think│    │Act  │    │Observe│
└──┬──┘    └─────┘    └──┬──┘
   │      ↑ 循环 ↑       │
   └──────────────────────┘
便利性↑↑ 但 确定性↓

MCP 统一协议：
┌──────────────────────────────┐
│      MCP = USB Hub           │
│  ┌────┐ ┌────┐ ┌────┐       │
│  │地图│ │搜索│ │数据库│ ...   │
│  └────┘ └────┘ └────┘       │
│  统一接口，即插即用            │
└──────────────────────────────┘

特性变化：便利性↑↑ 完备性↑ 确定性↓
成本变化：确认成本↑↑（自主决策→需要更多检查）

```

### 整体目标
展示自主调度的突破——便利性大幅提升，但代价是确定性下降。这是最关键的转折点。

### 串讲词
> "到了 v3.0，我们让模型自己想办法。ReAct——先思考、再行动、再观察、不够就再来一轮。便利性大幅提升，你不用一步步教了。"
>
> "MCP 呢，就像一个 USB Hub——所有工具统一接口，即插即用。不管是地图、搜索还是数据库，都走一个标准。"
>
> "但注意——确定性下降了！模型自己做了决策，你就要花更多时间去检查它做对了没有。确认成本上去了。"
>
> "相当于让应届生自己想办法干，但你要多花时间验收。"
