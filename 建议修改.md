# 课件审查建议

> 审查视角：企业内部员工（群众）+ 领导层
> 审查方法：费曼学习法（能否用最简单的话讲清楚？）+ 苏格拉底提问（听众会追问什么？）+ 波兰尼默会知识（那些"你知道但说不出来"的东西是否被点破？）

## 一、整体叙事结构评价

### 做得好的

1. **"装备升级"主线非常清晰**。绿装→蓝装→紫装→橙装的递进，让技术演进有了游戏化的记忆点，这是极好的教学设计。
2. **三个特性（确定性、完备性、便利性）作为评价框架反复出现**，给听众一个统一标尺，贯穿始终，费曼学习法的核心——"找到一个简单的类比，从头用到尾"——做到了。
3. **"应届生入职"的类比**（Slide05）非常精妙，把抽象的 LLM 弱具象化为人人都懂的职场场景。
4. **Slide18（这个课件本身）**是全场最佳设计之一——用自身作为案例证明人机协作，元叙事（meta-narrative）的效果极强。

### 需要思考的问题

#### 1. 缺少"为什么现在要听这个"的紧迫感

Slide01 列出了三个问题，但缺少一个关键维度——**为什么领导要我关心这个？**

从群众角度："这东西跟我日常工作有什么关系？"
从领导角度："这东西能为我的团队带来什么？投入产出比是多少？"

**建议**：在 Slide01 或 Slide02 之后，增加一张"为什么现在"的幻灯片，简要说明：
- 行业趋势（竞品/友商在做什么）
- 内部现状（我们有多少重复性工作可以被 Agent 辅助）
- 时间窗口（技术成熟度到了什么阶段）

#### 2. 从"三个好特性"到"应届生入职"之间的跳跃

Slide02 讲"三个好特性"（确定性/完备性/便利性），Slide03 讲"四种职业图鉴"，Slide04 讲"OpenClaw vs 大模型"，然后 Slide05 才是"应届生入职"。

这段叙事的问题是：听众在 Slide03-04 还没有理解"为什么大模型需要升级"，就被拉去对比了。应届生的类比很好，但来得太晚。

**建议**：
- Slide02 之后直接引入"应届生"类比，让听众先对大模型产生共情
- 然后再用"职业图鉴"说明"不同方案在这个应届生身上加了什么"
- 或者至少在 Slide03-04 之间加一句过渡："这四种职业，对应的是四种给应届生的培养方案"

#### 3. Slide04（OpenClaw vs 大模型）太早出现

此时听众还不知道 Prompt、RAG、ReAct、Skill 分别是什么，Slide04 就列出了"8件装备"。听众会困惑："这些装备名字我一个都没听过，你告诉我它们让三个特性都变好了？凭什么？"

**建议**：
- Slide04 移到 Slide10（v4.0）之后，作为"回头看"的总结页
- 或者把 Slide04 简化为仅展示"升级前后的属性对比"，不列出具体装备名称，等后续 slide 逐个展开

## 二、逐 Slide 详细建议

### Slide00 封面

**做得好**：简洁、信息明确。

**可改进**：
- 标题"从理解特性到快速上手"——但实际上讲完 Slide20，听众真的能"快速上手"吗？讲座更多是"理解"而非"上手"。建议改为"从理解原理到找到切入点"或直接去掉副标题。
- "中国移动内部培训"和"技术部"同时出现——如果听众来自不同部门，"技术部"标签可能产生距离感。

### Slide01 问题引入

**做得好**：背景标签云 → 三个核心问题的动态过渡很精彩。

**可改进**：
- 第三个问题"为什么用不起来"是最有共鸣的，但目前只占 1/3。建议将这个问题放大，因为它直击痛点。
- **苏格拉底提问**：听众心里会想——"你说的'用不起来'，具体指什么？能举一个我们公司内部的例子吗？"建议把第三个问题的 sub 改为一个具体的内部场景（比如"调了一个 Agent 处理工单，结果经常答非所问，最后还是人工处理"）。

### Slide02 三个好特性

**做得好**：三角图的可交互设计很棒。

**可改进**：
- "确定性、完备性、便利性"这三个词对非技术人员不够直观。
- **费曼学习法检验**：你能让一个行政部的同事在一句话内理解"确定性"吗？目前的定义是"系统输出结果可靠、可预测"——可以，但展开后太长了。
- **建议**：在三个特性旁边各加一个"大白话"版本：
  - 确定性 = "靠不靠谱"
  - 完备性 = "能干多少活"
  - 便利性 = "用起来省不省心"
- 三角关系图的"三个都想做好，需要持续投入"——**波兰尼的视角**：这其实就是经典的项目管理铁三角（快/好/省），但听众可能没有意识到这个类比。可以考虑点破。

### Slide03 职业图鉴

**做得好**：游戏化类比（职业加点）非常好，数值化对比直观。

**可改进**：
- "传统代码开发"和"大模型"是听众最容易理解的，但"传统机器学习"对非技术听众可能太抽象。建议简化为"统计模型/规则引擎"。
- 每个职业的"tagline"可以更口语化。比如"覆盖面最广，但不确定度极高"→"什么都能聊，但经常一本正经地胡说八道"。
- **缺失**：没有中国移动内部的对应案例。"传统代码"对应什么系统？"工作流"对应什么平台？听众需要看到自己身边的东西。

### Slide04 OpenClaw vs 大模型

**位置问题**（如上所述）：建议后移或简化。

**内容问题**：
- 8 件装备中的"MCP"和"Harness Eng."在 Slide04 出现时，听众完全不知道是什么。这种"提前剧透"反而会造成认知负担。
- **建议**：如果保留在此页，只展示装备的"小名"（如"工具调用""按需查知识""自主规划""经验沉淀"），不用技术术语。

### Slide05 应届生入职 + 三个核心困难

**做得极好**：清北应届生的类比是全场最佳设计之一。

**可改进**：
- Slide05 和 Slide06 内容有重叠。Slide05 讲"三个困难"（输入模糊/模型幻觉/注意力稀释），Slide06 又讲"大模型的三个核心困难"（意图澄清/模型幻觉/注意力稀释）。内容几乎一样。
- **建议**：合并为一张 slide，或者 Slide05 讲"类比"（应届生的困难），Slide06 讲"技术原理"（为什么大模型有这些困难），避免重复。
- **波兰尼视角**：这里缺了一个关键的默会知识——"为什么应届生/大模型会幻觉？因为它的'自信'不是来自'知道'，而是来自'看起来像知道的'"。这个洞察对理解大模型至关重要，但目前只是轻描淡写一句"本质是概率模型"。

### Slide07 v1.0 人工教学

**做得好**：Prompt 五要素的拆解非常实用，Function-call 的 JSON 示例真实。

**可改进**：
- **太技术了**。这是全场第一个"代码级"的内容。对非技术听众，Prompt 的五要素拆解可能过于详细。
- **费曼检验**：行政部的同事能看懂 Prompt 示例吗？如果能，说明讲得好。如果不能，需要加一层"人话翻译"。
- "Prompt 不能无限长"的局限说明很好，但可以更具体——比如"超过 X tokens 后，模型的遵从率下降 Y%"。
- **缺失**：没有讲"为什么 Function-call 比纯文字输出更可靠"的原理。听众会问："它为什么不直接告诉我答案，而是要调用工具？"——这个问题的答案（确定性工具 > 概率性生成）是理解后续所有"升级"的关键。

### Slide08 v2.0 流程标准化

**做得好**：RAG 四步演示是全场信息密度最高的交互设计之一，分步展开很清晰。知识图谱 RAG 的补充也很好。

**可改进**：
- **信息量太大**。RAG 有 4 步 × 每步有大量细节，知识图谱 RAG 又有 3 步。两个 Tab 加起来内容太多，一张 slide 放不下。
- **建议**：拆为两张 slide——RAG 一张，知识图谱 RAG 一张（或 Workflow 单独一张）。
- **苏格拉底提问**：听众会问——"RAG 检索到的资料如果是过时的怎么办？"这是一个真实的业务痛点，目前没有讨论。
- Workflow 部分目前只有知识图谱，缺少传统 Workflow（DAG/条件分支）的可视化。听众可能分不清"RAG"和"Workflow"的区别。

### Slide09 v3.0 自主调度

**做得好**：ReAct 八步播放的交互设计是全场最精彩的部分。MCP 的 USB Hub 类比非常直观。

**可改进**：
- ReAct 演示中的步骤偏多（8步），听众到第 5-6 步时注意力可能下降。建议保留核心的"思考→行动→观察→发现不足→再思考→再行动"循环，把 8 步精简为 6 步。
- MCP 的代码示例（高德地图）很好，但 14 个工具的列表展示太密集。建议只展示 3-4 个代表性工具，标注"完整列表有 14 个"。
- **确定性↓的标注非常诚实**。但这引出一个关键问题："确定性下降了，那还要不要用？"——建议在这里加一句过渡："确定性下降了，所以我们需要 v4.0 来补回来。"

### Slide10 v4.0 龙虾！

**做得好**：SKILL.md 的真实格式展示、Harness 的"烈马"类比、对比表。

**可改进**：
- "龙虾"这个称呼突然出现，之前没有任何铺垫。如果这是内部代号，建议在 Slide04 或更早的地方就提一下"我们内部叫它龙虾"。
- Harness 的对比表（传统 Prompt Engineering vs Harness Engineering）很好，但最后一行"Channel：多端触达与消息路由"——这在之前没有任何铺垫，听众不知道"Channel"是什么。
- "自我进化能力"部分是展开折叠的，容易被忽略。建议改为默认展开，因为这是 Harness 的核心卖点。
- **波兰尼视角**：Harness 的本质是"把人的默会知识显性化"——这个洞察没有点破。建议在 Harness 的描述中加一句："很多你'说不清楚但就是会做'的东西，Harness 帮你说清楚，然后让 AI 照着做。"

### Slide11 架构全景

**做得好**：四层堆叠 + 点击展开的交互设计。

**可改进**：
- 这是全场最"架构师视角"的 slide。对普通员工来说，"Gateway""Context Assembly""ReAct Loop"这些词太专业了。
- **建议**：每层加一个"人话版本"：
  - Gateway = "前台接待"（所有请求先到这里）
  - 认知层 = "大脑记忆区"（记住你是谁、会什么）
  - 执行层 = "手脚"（实际干活的地方）
  - 触达层 = "通讯工具"（飞书/企微/钉钉）
- "200 个 Skill 仅占 ~4.8k tokens"这个数据点很好，但可以更具体——对比一下"如果全量注入会占多少 tokens"。

### Slide12 四个使用层级

**做得好**：层级划分清晰，适用场景标注实用。

**可改进**：
- "不要一开始就追求 Level 4"——这句话很好，但**建议加一个"推荐路径"**：从哪个层级开始、如何升级、升级的标志是什么。
- **缺失**：每个层级的"投入时间"参考。比如 Level 1 需要多久能上手？Level 3 需要多久的调教期？
- **领导视角**：缺少 ROI（投入产出比）的讨论。"我投一个人月，能节省多少人力？"这是领导最关心的问题，目前完全没有涉及。

### Slide13 局限性

**做得好**：诚实地讨论局限性，Token 消耗的动态演示非常直观。"用户说了 2 个字，消耗了 13,500 tokens"——这个结论很有冲击力。

**可改进**：
- "截图/动画占位"的虚线框——应该填充真实内容或删除。目前"模型幻觉"和"安全边界"展开后只显示"截图/动画占位"，会显得不完整。
- **苏格拉底提问**：听众会问——"那我们怎么控制成本？"建议加一个"应对策略"的小节，哪怕是 2-3 条建议（如：先用 Level 1 验证价值再升级；关键流程用确定性脚本；控制 Skill 数量避免过度加载）。
- "公网 Key 价格高，模型能力有限"——这句话在内部培训中可能需要更具体的数字。中国移动内部是否有自己的模型服务？成本对比如何？

### Slide14 怎么做好龙虾

**做得好**：适用/不适用场景的对比非常实用。

**可改进**：
- 标题"怎么做好龙虾"——游戏化可以，但这一页的内容其实非常严肃（策略选择）。建议在标题和内容之间加一个过渡句。
- "Agent 的上限取决于你的耐心和标准"——这句金句很好，但放在"适用场景"页显得有些突兀。建议移到 Slide20 总结。
- **缺失**：没有讨论"如何评估 Agent 的效果"。比如：如何衡量确定性？如何度量完备性？如何判断便利性提升了？

### Slide15 面对复杂问题

**做得好**：四条进化路线的对比、属性值变化。

**可改进**：
- "给伙伴们也氪金"——这个比喻对非游戏玩家可能不直观。
- Claude Code / Codex、DeerFlow / Dify、Evolver / EvoMap、OpenClaw / Hermes——这些产品名太多了。听众分不清哪些是中国移动内部的，哪些是外部的。

- **关键缺失**：没有讨论"从哪个路线开始"。如果我是运维人员，我应该先从哪个升级路线切入？

### Slide16 浏览器脚本自动化

**做得好**：六步闭环的人机协作展示。

**可改进**：
- **这不是一个中国移动内部的典型场景**。"浏览器脚本自动化"对运营商员工来说比较远。
- **建议**：改为更贴近内部的场景，比如：
  - 客服工单自动分类 + 回复建议
  - 网络告警自动分析 + 处理建议
  - 日报/周报自动生成
- "人负责'做什么'和'够不够好'，AI 负责'怎么做'和'做多少次'"——这句话极好，但被埋在了底部。建议提升到 slide 的开头位置。

### Slide17 团队协作模式

**做得好**：多角色协作的概念模型。

**可改进**：
- "此处可替换为团队实际使用截图或 GIF"——**必须替换**。目前是一个占位符。
- 团队协作的流程描述太抽象。建议用一个具体的内部案例（如：网优团队 + Agent 的日常协作流程）。
- **缺失**：没有讨论"团队中谁负责 Agent 的调教和维护"。这是领导最关心的组织问题——"我需要安排谁来管这个 Agent？"

### Slide18 这个课件本身

**做得极好**：元叙事的运用——用课件本身证明人机协作的效果，比任何理论都有说服力。

**可改进**：
- "你可以外包你的思考，但不能外包你的理解"——这句话是全场最重要的金句之一，但它被放在了底部。建议把它提升到更醒目的位置。
- 协作分工（人负责/ AI 负责/ 人+AI 共同）的展示很好，但缺少一个**时间维度**："这个过程花了多长时间？如果纯手工需要多长时间？"——时间对比是最有说服力的 ROI 数据。

### Slide19 未来展望

**做得好**：已有/建设中/未来的三段式结构清晰。

**可改进**：
- "聚智平台""磐匠 Agent""云虾"——这些内部项目名对外部门同事可能不熟悉。建议加一句话描述每个项目是什么。
- **领导视角**：缺少**时间线**和**里程碑**。领导想知道的不是"未来要做什么"，而是"什么时候能做到什么"。
- "Agent 不会取代人，但会用 Agent 的人会取代不会用的人"——这句话有争议性，可能引起部分听众的焦虑。如果使用，建议加一句安抚："我们的目标不是淘汰，而是赋能。"

### Slide20 总结

**做得好**：三条 take-away 简洁有力。

**可改进**：
- 总结应该**回应 Slide01 的三个问题**。Slide01 提了三个问题，但总结没有显式地回答它们。建议加一个"问题回顾"：
  - Q: "这些概念都是什么？" → A: "它们是给大模型穿上的 8 件装备"
  - Q: "OpenClaw 是什么？" → A: "穿满装备的大模型 + 驾驭术"
  - Q: "为什么用不起来？" → A: "因为没有理解边界和投入调教"
- **缺失**：没有"行动号召"（Call to Action）。听完讲座后，听众应该做什么？建议加一句："从今天开始，用 Level 1 试一个小场景。"

### Slide21 谢谢

**做得好**：简洁。

**可改进**：
- 可以加一个二维码或链接，指向"OpenClaw 入门文档"或"内部 Agent 社区"。

## 三、UI/交互设计建议

### 优点

1. **点击交互设计**：大部分 slide 都有分阶段展示（点击展开/播放），这对演讲节奏的控制非常好。
2. **配色系统**：绿/蓝/紫/橙四个"装备等级"的颜色贯穿始终，视觉一致性极强。
3. **动画节奏**：GSAP 的动画流畅，不会分散注意力。
4. **主题切换**：蓝/暗/暖三套主题对投影环境适配很好。

### 可改进

1. **Slide08（RAG）信息密度过高**——需要拆分或增加导航提示。
2. **代码块的可读性**：深色背景上的代码在投影环境下可能看不清。建议增大字号或提供"简洁版"。
3. **移动端体验**：部分 slide 在手机上的布局会挤压。如果听众可能在手机上回顾课件，需要测试移动端适配。
4. **进度指示**：底部的"点击查看/点击返回"提示很好，但建议加一个 slide 序号（如"7/21"），让听众知道进度。
5. **占位符**：Slide13 和 Slide17 有明显的"截图/动画占位"虚线框，需要在正式演示前填充。

## 五、讲解节奏建议

### 讲得太啰嗦的地方

1. **Slide02（三个好特性）**：三角图展开 + 详细定义 + 示例标签，内容太多。建议拆为"概览"和"详解"两个阶段，或缩减展开后的内容。
2. **Slide08（RAG）**：两个 Tab × 多步交互，如果演讲者每步都展开，可能要讲 10 分钟以上。建议精简为核心步骤（文档分块 + 相似度检索），略过向量化细节。
3. **Slide10（Skills + Harness）**：SKILL.md 的完整示例太长。建议只展示 key 部分（frontmatter + 处理流程的前两步）。

### 讲得太简单的地方

1. **"大模型为什么会幻觉"**——目前只有一句"本质是概率模型"。这是理解整个讲座的基础，建议用 2-3 句话讲透。
2. **"为什么 Function-call 比自然语言更可靠"**——这是 v1.0 的核心价值，但目前没有解释原因。
3. **"确定性到底怎么提升"**——Slide02 定义了确定性，Slide07-10 展示了各种手段，但从未解释"为什么这些手段能提升确定性"的底层逻辑。
4. **"Harvest/Harness 的区别"**——Harness 对比了传统 Prompt Engineering，但没有解释"为什么 Harness 能做到传统方法做不到的事"。

## 六、总结

### 这个课件最核心的价值

它成功地把一个复杂的技术体系（Agent/RAG/ReAct/Skill/Harness）翻译成了一个非技术人员能理解的叙事（应届生入职 → 穿装备升级 → 龙虾诞生）。这个翻译工作的质量很高。

### 最需要补的一个东西

**内部案例**。目前所有的示例（信号覆盖、5G-A、浏览器脚本、高德地图）都是虚构的或通用的。中国移动的听众需要看到一个"我们公司内部的 XX 团队，用 Agent 做了 XX，节省了 XX 时间"的真实故事。一个真实案例 > 十个理论框架。

### 最需要删/简化的一个东西

**Slide05 和 Slide06 的内容重叠**。两页讲的是同一件事（大模型的三个困难），合并后叙事会更紧凑。

### 最重要的金句（已有，建议强化位置）

1. "你可以外包你的思考，但不能外包你的理解。"（Slide18）
2. "人负责'做什么'和'够不够好'，AI 负责'怎么做'和'做多少次'。"（Slide16）
3. "不是每个职业都需要升到满级。找准自己的场景，氪对的金，比全氪更高效。"（Slide15）

这三句话应该在讲座的不同阶段至少各出现两次，帮助听众记住核心信息。
